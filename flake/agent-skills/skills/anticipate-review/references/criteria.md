# レビュー観点

対象レビュアーのレビューコメントから抽象化した観点集。
指摘スタイルの傾向: 断定より「〜はどうなんですかね…？」「〜の方がいいと思うんですが」という問いかけ・提案形式が多い。
理由や参照先（公式ドキュメント・設計原則）をセットで示すことが多い。

> **データソース**: 1000 件規模のレビューコメントから抽出（2021–2026 年）

### C-001: 既存パターンとの整合性 {#c-001}

**重要度**: Medium

**シグナル**:
diff に追加・変更されたコードのスタイルや設計が、同一リポジトリの既存コードと大きく異なっている。特に API 設計、ページネーション方式、エラー処理パターン、テストの書き方が既存と食い違っている。

**なぜ問題か**:
「既存の実装があるのであればそちらに統一した方がいい」という傾向が強い。一貫性のないパターンは読み手の理解を妨げ、将来の一括変換時に取りこぼしが起きやすい。

**チェック方法**:

1. diff で追加されたコードと、同ディレクトリ・同パッケージ内の既存コードを比較する
2. 特に API のページネーション方式・エラーレスポンス形式・命名規則が揃っているか確認する
3. PR 説明に「意図的に変えた理由」が書かれていれば指摘しない

**指摘テンプレ（問いかけ形式）**:
既存の `<類似ファイル/テスト>` では `<X>` という方式を使っていますが、この変更では `<Y>` になっています。既存に合わせる形ではいかがでしょうか？

### C-002: lint / 静的解析の例外設定 {#c-002}

**重要度**: Medium

**シグナル**:
diff に `//nolint`, `#nosec`, `// eslint-disable`, `/* eslint-disable-next-line */` 等の lint 抑制コメントが追加されている。

**なぜ問題か**:
「この lint に例外は設定しない方がいいと思います」というスタイルが確認されている。例外を設定するより、lint が指摘している根本原因を修正する方を好む。例外が増えると lint の価値が下がり、真の問題を隠す恐れがある。

**チェック方法**:

1. diff に lint 抑制コメントが含まれるか確認する
2. 含まれる場合、その場所のコードを見てなぜ lint が警告しているか理解する
3. 例外設定が正当化できるか（外部ライブラリの型制約など、コードで対処できない理由）を判断する

**指摘テンプレ（問いかけ形式）**:
`<場所>` に lint 例外が設定されていますが、`<lint ルール名>` が警告している原因をコード側で修正することはできますか？例外を増やすより根本原因の対処を検討してもらえると助かります。

### C-003: 将来の変更漏れリスク {#c-003}

**重要度**: High

**シグナル**:
diff が「すべての新規追加に対して手動で適用しなければならない横断的な仕組み」を導入している。例: gRPC インターセプタ、認証ミドルウェア、ロールベースの認可、CI ステップ。

**なぜ問題か**:
「stream の RPC を足したときに認証をかけ忘れるリスクを考えると、はじめから追加しておいた方がいい」という指摘パターンが確認されている。手動で毎回適用する設計は、将来の追加時に適用漏れが起きやすい。

**チェック方法**:

1. 「新しい〜を追加するたびに〇〇もしなければならない」という構造になっていないか確認する
2. 認証・認可・ロギング・メトリクスなど横断的関心事が手動適用になっていないか見る
3. 自動で適用される仕組み（デフォルトインターセプタ、ベースクラス、テンプレート等）に変更できるか検討する

**指摘テンプレ（問いかけ形式）**:
`<仕組み>` を追加するとき、将来 `<新しいX>` を追加するたびに手動で同じ設定が必要になります。うっかり忘れるリスクを考えると、デフォルトで適用される形に変えることはできますか？

### C-004: エラー・ログ出力の分類 {#c-004}

**重要度**: Low

**シグナル**:
diff のエラーハンドリングで、ユーザーが対処できるエラーと開発者向けのデバッグ情報が同じ出力経路（`log.Printf` or `fmt.Printf` で統一）になっている。

**なぜ問題か**:
「ユーザーの操作で修正できる範囲は `fmt.Printf` で出力して、開発者が修正するための情報を `log.Printf` で出した方がいい」というスタイルが確認されている。混在していると、エンドユーザーに不要なスタックトレースが出たり、逆に開発者が必要な情報を得られなかったりする。

**チェック方法**:

1. diff のエラーハンドリングで `fmt.Printf` / `log.Printf` / `fmt.Errorf` が混在していないか確認する
2. ユーザー向けメッセージ（操作ミス、入力エラー）と内部エラー（バグ、システム障害）が適切に分かれているか見る

**指摘テンプレ（問いかけ形式）**:
`<場所>` のエラーはユーザーが対処できるものですか、それとも開発者向けのデバッグ情報ですか？前者なら `fmt.Printf`、後者なら `log` パッケージで出すとエンドユーザーに余分な情報が出なくなります。

### C-005: コンテナイメージタグの固定性 {#c-005}

**重要度**: Medium

**シグナル**:
diff に Docker イメージ参照が含まれ、タグが `:latest` や `v1.2.3` のような可変タグになっている。

**なぜ問題か**:
「タグはコミットハッシュか git tag (commit hash) がいい」というスタイルが確認されている。可変タグはデプロイの再現性を損ない、何がデプロイされているか追跡しにくくなる。

**チェック方法**:

1. diff にコンテナイメージ参照が含まれるか確認する（Dockerfile、k8s マニフェスト、CI 設定など）
2. `:latest` や浮動バージョンタグが使われていたら指摘する

**指摘テンプレ（問いかけ形式）**:
`<場所>` のイメージタグが `<:latest などの可変タグ>` になっていますが、コミットハッシュ固定にするとデプロイの再現性が上がります。検討いただけますか？
