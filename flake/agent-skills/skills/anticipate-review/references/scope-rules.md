# スコープ判定ルール

team-review を適用すべき diff かどうかを判定するルール。

## オペレーション系 PR の定義

以下の特徴を持つ変更は「オペレーション系」とみなし、観点照合をスキップする。

### パターン 1: リソース削除・整理

diff 内の変更が以下のみで構成される:

- 設定ファイルから特定リソースのエントリを削除
- インフラリソースの参照を削除（構成管理ツールの inventory、監視ツールの targets 等）
- コードロジックの変更がない（追加行がほぼなく、削除行のみ、または設定値の変更のみ）

### パターン 2: 自動更新系

diff 内の変更ファイルが以下のみ:

- `*.lock` 系ファイル全般（言語・ツール別の lock ファイル）
- `renovate.json` / 依存関係バージョン番号の更新のみ
- 自動生成コード（`*generated*` 等の命名規則を持つファイル）の更新のみ

### パターン 3: PR タイトル・コミットメッセージのパターン

以下の Conventional Commits プレフィックスのみで構成される PR は、コードの評価より変更内容の検証が主目的であることが多く、観点照合の効果が限定的:

- `chore:` のみ（`chore: update deps`、`chore: bump version` 等）

> **注意**: `refactor:`, `feat:`, `fix:`, `test:` は観点照合の対象。`chore:` でも実装変更を含む場合は通常レビュー対象。

## 判定フロー

```
1. git diff --name-only でファイル一覧を確認
   │
   ├── 変更ファイルがすべて *.lock / *generated* / renovate.json ?
   │   → 「ロックファイル・自動更新のみの変更のため、観点照合をスキップします」で終了
   │
   ├── 変更行がすべて削除行（- のみ）で追加行（+）が設定値変更のみ?
   │   → 「オペレーション系の変更のため、観点照合をスキップします」で終了
   │
   └── 上記以外 → 通常レビュー対象（SKILL.md の Step 3 へ）
```

## オペレーション系判定時の出力例

```
## team-review: スコープ判定結果

**判定**: オペレーション系の変更

**理由**: 差分がリソース削除エントリのみで構成されており、コードロジックの変更がありません。

このタイプの変更は変更内容の正確性（正しいリソースを削除しているか等）の確認が主目的のため、
コードレビュー観点の照合はスキップします。
```

## 混在している場合

オペレーション系変更とコード変更が混在している場合は、コード変更の部分のみを対象に観点照合を行う。混在していることを出力の冒頭で明示する。
